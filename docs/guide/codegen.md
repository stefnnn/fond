# Codegen

`bin/rails fond:codegen` eager-loads the app, walks every registered page's
`Params`/`Props`, every mutation's `Params`/`Props`, and (if configured)
the shared props struct, and writes four files to `Fond.config.output_dir`
(default `app/frontend/generated/`):

| File | Contents |
| --- | --- |
| `types.ts` | `interface`s for every `T::Struct` reached, `type` literal unions for every `T::Enum` |
| `pages.ts` | `pages` map of `component_name` → dynamic `import()`, for the router's code-splitting |
| `hooks.ts` | `use<Page>()`, `use<Mutation>()`, and `useShared()` (if configured) |
| `paths.ts` | `paths.<page>(params)` — typed URL builders |

All four start with the same header and are meant to be committed:

```ts
// Generated by fond. Do not edit.
// Regenerate with: bin/rails fond:codegen
```

## `types.ts`

Excerpt from `examples/orders` (`app/frontend/generated/types.ts`):

```ts
export type ISODateTime = string;

export type OrderStatus = "pending" | "paid" | "shipped" | "cancelled";

export interface LineItemDTO {
  id: number;
  productName: string;
  quantity: number;
  unitPriceCents: number;
}

export interface NoteEventDTO {
  type: "note";
  id: number;
  body: string;
  author: string;
  createdAt: ISODateTime;
}

export interface OrderDTO {
  id: number;
  customerName: string;
  customerEmail: string;
  status: OrderStatus;
  totalCents: number;
  placedAt: ISODateTime;
  notes: string | null;
}
```

Every reachable struct/enum is discovered transitively (a breadth-first
walk starting from every page's `Params`/`Props`, every mutation's
`Params`/`Props`, and the shared props class), then emitted sorted by
name, so the file diffs cleanly in git regardless of discovery order.

### Type mapping

| Ruby (`type_object`) | TypeScript |
| --- | --- |
| `Integer`, `Float`, `Numeric` | `number` |
| `String`, `Symbol` | `string` |
| `T::Boolean` | `boolean` |
| `NilClass` | `null` |
| `Date` | `ISODate` (branded `string` alias, emitted only if used) |
| `Time`, `DateTime` | `ISODateTime` (branded `string` alias, emitted only if used) |
| `T::Struct` subclass | the struct's generated interface name |
| `T::Enum` subclass | a literal union of `.serialize` values, e.g. `"pending" \| "paid" \| ...` |
| `T::Array[X]`, `T::Set[X]` | `X[]` (parenthesized if `X` is a union) |
| `T::Hash[K, V]` | `Record<K, V>` |
| fixed-shape hash | `{ key: type; ... }` |
| `T.nilable(X)` | `X \| null` |
| `T.any(A, B, ...)` | `A \| B \| ...`, deduped and sorted; `T.any(TrueClass, FalseClass)` collapses to `boolean` |

Wire format for temporal types is fixed globally, not per-field: `Date` →
`iso8601`, `Time`/`DateTime` → UTC `iso8601(3)` (millisecond precision) —
see `Fond::Serialize.to_wire`. The `ISODate`/`ISODateTime` aliases exist so
the type signals intent even though both are plain strings on the wire.

### Discriminated unions

Sorbet has no literal types, so a `T.any` of structs can't express a
discriminant in Ruby's type system. The convention: give each member a
`type` prop with a `String` default acting as the tag —

```ruby
class StatusChangeEventDTO < T::Struct
  const :type, String, default: "status_change"
  # ...
end

class NoteEventDTO < T::Struct
  const :type, String, default: "note"
  # ...
end

OrderEventDTO = T.type_alias { T.any(StatusChangeEventDTO, NoteEventDTO) }
```

Codegen special-cases exactly this shape: a prop literally named `type`
whose `type_object` is `String` and whose `default` is a `String` is
emitted as the literal (`type: "note";`) instead of widening to `string`.
The result is a discriminated union TypeScript can narrow on:

```ts
export interface NoteEventDTO {
  type: "note";
  // ...
}
export interface StatusChangeEventDTO {
  type: "status_change";
  // ...
}
// activity: (NoteEventDTO | StatusChangeEventDTO)[]
```

`Fond::Coerce` reads the same default-as-literal convention on the way in
(see [Pages & DTOs](/guide/pages#coercion-rules)) — codegen and the
runtime coercer agree on what "discriminated union" means because they're
both reading `prop[:default]` off the same struct.

## `hooks.ts`

One `use<Name>()` per page (returns `Props`, resolved synchronously from
the current page payload — it's not a fetch), one `use<Name>()` per
mutation (returns a `Mutation<Params, Props | null>`), and `useShared()`
if `shared_props_class_name` is configured:

```ts
import { useMutation, usePageProps, useSharedProps, type Mutation } from "fond";
import type { OrdersIndexProps, OrdersShowProps, SharedProps, /* ... */ } from "./types";

export function useOrdersIndex(): OrdersIndexProps {
  return usePageProps<OrdersIndexProps>("orders/index");
}

export function useShared(): SharedProps {
  return useSharedProps<SharedProps>();
}

export function useOrdersUpdateStatus(): Mutation<OrdersUpdateStatusParams, null> {
  return useMutation<OrdersUpdateStatusParams, null>("/orders/:id/update_status", "patch", ["id"]);
}
```

The mutation hook's arguments come straight from route binding: the path
pattern, the HTTP verb, and which of the `Params` fields are path
parameters (so `useMutation` knows to interpolate them into the URL
instead of the JSON body — see [Mutations](/guide/mutations)).

## `paths.ts`

```ts
import { buildPath } from "fond";
import type { OrdersIndexParams, OrdersShowParams } from "./types";

export const paths = {
  ordersIndex: (params?: Partial<OrdersIndexParams>) =>
    buildPath("/orders", [], params),
  ordersShow: (params: Pick<OrdersShowParams, "id"> & Partial<Omit<OrdersShowParams, "id">>) =>
    buildPath("/orders/:id", ["id"], params),
};
```

Signature depends on whether the page has required path params: none →
optional `Partial<Params>` (everything becomes a query string); some →
those keys required, the rest optional. If a page has no `Params` at all
and no path params, the entry takes no arguments.

## Naming

`name_for(klass)` builds the TypeScript name from the Ruby class's
namespace segments, stripping a trailing `Page`/`Mutation` off each
segment and joining them: `Orders::IndexPage` → `OrdersIndex`,
`Orders::CreateMutation` → `OrdersCreate`. A nested struct like
`Orders::CreateMutation::LineItemInput` → `OrdersCreateLineItemInput`.
Two different classes producing the same TS name is a codegen error (raised
at generation time, not silently overwritten).

Hook names are `use` + that name (`useOrdersIndex`, `useOrdersCreate`).
Path keys camelize `component_name` segments (`"orders/index"` →
`ordersIndex`). Field names are `Fond::Naming.camelize`d snake_case.

## Config

```ruby
Fond.configure do |config|
  config.output_dir = "app/frontend/generated"     # default
  config.pages_import_prefix = "../pages/"          # default; used in pages.ts import()s
  config.shared_props_class_name = "SharedProps"     # default nil — opts into useShared()
  config.version = -> { "dev" }                      # default; see the protocol's version mismatch handling
  config.ssr_url = nil                                # see SSR
  config.ssr_timeout = 1.0
  config.autogenerate = true                          # dev-mode codegen on code reload
  config.autogenerate_dirs = %w[app/pages app/mutations]
end
```

## Automatic regeneration in development

With `autogenerate` on (the default), fond regenerates the output on every
Rails code reload: it eager-loads `autogenerate_dirs`, prunes classes
orphaned by the reload, and rewrites only files whose content changed.
Editing a `Props` struct and refreshing the browser is enough — no manual
codegen step. Production and test environments never autogenerate; CI
should run the drift check.

One rule to keep reloading healthy: **one constant per file** (standard
Zeitwerk convention). A DTO file defining two structs will appear to work
until the first reload, then fail with a sorbet "redefine prop" error,
because Zeitwerk only unloads the constant matching the file name.

`pages_import_prefix` must match wherever your entrypoint's
`import.meta.glob` is rooted, since `pages.ts` literally emits
`import(${prefix}${component_name})`.

## Drift check

```bash
bin/rails fond:codegen:check
```

Regenerates in memory and compares byte-for-byte against the committed
files; aborts (non-zero exit) on any difference. Run this in CI — it's the
guarantee that what's committed is what the current Ruby definitions
actually produce, so a page/DTO change without a re-run of
`fond:codegen` fails the build instead of silently drifting.
